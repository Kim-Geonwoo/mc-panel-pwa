# 커밋 메시지 작성 스킬

이 스킬은 git 변경사항과 프로젝트 맥락을 분석해 커밋 제목과 메시지를 작성합니다.

## 실행 순서

### 1단계 — git 상태 파악 (병렬 실행)

아래 세 명령을 동시에 실행합니다.

- `git diff HEAD --stat` — 변경된 파일 목록과 규모
- `git diff HEAD` — 실제 변경 내용 (diff)
- `git log --oneline -5` — 최근 커밋 스타일 파악

스테이징된 변경이 없으면 `git diff --cached --stat` / `git diff --cached`로 대신 확인합니다.

### 2단계 — 변경 내용 분류

diff를 읽고 각 변경을 아래 유형 중 하나로 분류합니다.

| 유형 | 설명 |
| --- | --- |
| `feat` | 새 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 동작 변경 없는 코드 개선 |
| `docs` | 문서·주석만 변경 |
| `chore` | 빌드·설정·의존성 |
| `style` | 포맷·오타 수정 |
| `perf` | 성능 개선 |

변경이 두 유형 이상이면 비중이 큰 쪽을 앞에 씁니다. 예: `fix/docs:`, `feat/refactor:`

### 3단계 — 커밋 제목 작성

- 형식: `<유형>: <한 줄 요약>`
- 50자 이내
- 명사형 또는 동사 원형으로 끝냄 ("추가", "수정", "제거")
- 범위가 특정 모듈에 한정되면 `fix(api):` 형태로 표기 가능

### 4단계 — 커밋 메시지 본문 작성

본문에는 **왜** 변경했는지를 중심으로 씁니다. 형식은 아래를 따릅니다.

```
<제목>

<변경사항 1 — 무엇을, 왜>
<변경사항 2 — 무엇을, 왜>
...

<필요한 경우: 영향 범위, 주의사항, 관련 이슈>
```

규칙:
- 각 항목은 한 문장으로 요약
- "~을 수정함" 대신 "~가 ~해서 ~으로 수정" — 이유를 포함
- 파일명·함수명은 백틱으로 감쌈 (`handleChat`, `genSID`)
- 관련 없는 변경이 섞여 있으면 그룹별로 단락을 나눔

### 5단계 — 출력

아래 형식으로 최종 출력합니다. 코드 블록으로 감싸 복사하기 쉽게 합니다.

```
**커밋 제목**
\`\`\`
<제목>
\`\`\`

**커밋 메시지**
\`\`\`
<본문>
\`\`\`
```

## 주의사항

- 실제 커밋(`git commit`)은 실행하지 않습니다. 텍스트만 작성합니다.
- 변경 내용이 없으면 "커밋할 변경사항이 없습니다"라고 안내합니다.
- 민감한 파일(`.env`, credentials 등)이 포함되어 있으면 커밋 전 경고합니다.
- 최근 커밋 스타일(한국어/영어, 형식)을 파악해 일관성을 유지합니다.
